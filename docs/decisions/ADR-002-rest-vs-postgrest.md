# ADR-002 — Чистый REST `/api/v1/...` вместо PostgREST-совместимости

**Status:** Accepted
**Date:** 2026-05-25
**Deciders:** owner

## Контекст

Текущий бэк (v1) реализует **PostgREST-совместимый слой**: `GET /rest/v1/orders?status=eq.open&order=created_at.desc`. Фронт использует `supabase-js`, который под капотом строит именно такие URL. Это позволяет работать одинаково с локальным Electron-бэком и с облачным Supabase без правок фронта.

При переходе на v4 (Go-бэк) вопрос: продолжить ли эмулировать PostgREST?

Варианты:
1. **PostgREST-совместимый Go-бэк.** Фронт не трогаем.
2. **Чистый REST `/api/v1/<resource>`.** Удобнее, но фронт переписываем.

## Решение

**Чистый REST `/api/v1/...`. Фронт переписываем на отдельной фазе (см. [10-FRONTEND-ADAPTER.md](../prd/10-FRONTEND-ADAPTER.md)).**

## Обоснование

### Почему НЕ PostgREST

1. **PostgREST-стиль убивает бизнес-логику.** В нём нет «specialized endpoints»: всё — это `INSERT/UPDATE` строки. `close_order` со сложной логикой (revenue entry, deduct stock, print receipt, attach to shift) либо размазывается на 5 запросов с фронта, либо ломает паттерн через rpc-функции. У нас уже **сейчас** проблема: бизнес-логика дублируется на фронте — это прямое следствие PostgREST-стиля.
2. **Трудно валидировать.** На PostgREST request body — это сырая строка таблицы. Нет DTO с validator-тегами, нет специфичных ошибок типа `STOCK_INSUFFICIENT`. Получаем «PG error: check constraint violated» — UX страдает.
3. **Сложно версионировать.** REST-эндпоинты можно версионировать (`/api/v1`, `/api/v2`). PostgREST-схема — это сама БД, изменения ломают клиентов.
4. **Поддерживать в Go дороже.** Парсер фильтров `eq.`, `neq.`, `like.`, `in.`, `or=(...)` — это ~300 строк кода, который мы будем чинить вечно. Чистый REST — `chi.URLParam("id")`, всё.
5. **Хочется ясной OpenAPI-спеки.** На REST один YAML описывает контракт; фронт авто-генерирует TS-типы. На PostgREST контракт неявный, типы фронта — самописные.
6. **Логи и метрики:** `POST /api/v1/orders/{id}/close` в графиках — понятно. `POST /rest/v1/orders` с body — что произошло, надо смотреть payload.

### Почему НЕ важна совместимость с супабейс-фронтом

Аргумент за PostgREST — «не нужно переписывать фронт». Но:
- Бизнес-логика во фронте всё равно требует рефакторинга (поэтому Phase 2 — фронт-адаптер).
- React Query + кастомный API-клиент быстрее, удобнее и типобезопаснее, чем `supabase-js`.
- Auto-gen TS-типов из OpenAPI убирает руками поддерживаемые `lib/types.ts`.

То есть «фронт не трогаем» — иллюзия. Трогать всё равно придётся. Лучше трогать **разумно**.

### Кому пожертвовали

Пожертвовали возможностью «бесшовно вернуться на Supabase, переключив URL». Это была реальная фича в v1. Но:
- В v4 cloud не используется (см. [07-FUTURE-CLOUD.md](../prd/07-FUTURE-CLOUD.md)).
- Если когда-то вернёмся в cloud — это будет наш Go-бэк, развёрнутый в облаке, с тем же REST. Supabase-клиент не нужен.

## Последствия

### Положительные

- Бизнес-логика на бэке, фронт чище.
- Один OpenAPI = типы + контракт + Swagger UI = удобно для отладки.
- Specialized endpoints (`close`, `split`, `request-bill`) явные.
- Валидация и ошибки качественные.
- Метрики и логи читаемые.
- Версионирование через `/api/v1`.

### Отрицательные

- Фаза 2 переписывания фронта (~9 недель) — реальная стоимость.
- Параллельный режим (v1 Node + v4 Go) сложнее: фронт смотрит то туда, то сюда → нужны feature-флаги.

## Альтернативы (отвергнуты)

### Гибрид: PostgREST для read, REST для write

Рассматривался: оставить `GET /rest/v1/orders` для совместимости, новые мутации делать через REST. Отвергнуто — двойной mental model, double maintenance, и read-эндпоинты всё равно хочется писать с фильтрами/preload.

### tRPC

Альтернатива OpenAPI + REST для TS-фронта. Отвергнуто — Go-бэк, tRPC привязан к Node. Если бы бэк был на Node — рассматривали бы серьёзно.

### GraphQL

Отвергнуто — overkill для нашего scope. Один тип клиента (наш React), одно API. GraphQL даёт гибкость, не нужную здесь, и удорожает сервер.

## Открытые вопросы

- Соглашение по фильтрам выбрано Django-style (`?status__in=open,closed`). Альтернатива — JSON:API стиль (`?filter[status]=open`). Менее популярна в нашем стеке, ушли от.
- Pagination: `?limit=&offset=` vs cursor-based. Для MVP — offset. Cursor — если столкнёмся с большими таблицами (audit_log на отчётах).
