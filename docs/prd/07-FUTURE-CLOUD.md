# 07 — Future Cloud (вне scope v4)

В v4 **cloud не делаем**. Всё локально на машине кассира: Go-бэк + embedded PostgreSQL. Этот документ — заметка на будущее, когда захочется вернуть облачное зеркало для Owner Dashboard / multi-location-аналитики.

## Когда это понадобится

- Появилось >1 ресторана у одного владельца → нужен сводный отчёт.
- Owner хочет смотреть выручку из дома → ему нужен Web Dashboard, а Web Dashboard'у нужны данные где-то в облаке.
- Появилась сеть из >5 точек → нужна центральная аналитика, mobile-приложение для владельца.

Пока всего этого нет — v4 живёт автономно, без интернета.

## Архитектурные варианты (на будущее)

### Вариант A — One-way push в облако

Тот же подход, что был в v1: локальный Postgres = source of truth, отдельный worker асинхронно пушит дельты в облако.

```
[Restaurant local Postgres] ──(push job)──> [Cloud Postgres / Supabase] ──> [Owner Dashboard]
                                                  read-only зеркало
```

- Что переиспользуем из v1: логику FK self-healing из `desktop/sync.js` (CHILD_PARENT_MAP).
- Что добавляем: `sync_log` таблица (триггеры или GORM-хуки на каждой tracked-table), worker в `internal/cloud/pusher.go`, конфиг `cloud.enabled: true/false`.
- Cloud-БД: Supabase или собственный managed Postgres + PostgREST.

### Вариант B — Полное двустороннее облако

Бэк живёт только в облаке, локально остаётся только тонкий cache + offline-очередь. Это уже **архитектурное переписывание**, не доработка v4. Если поедем туда — это будет v5.

### Вариант C — Edge-агрегатор

Бэк локально + центральная headquarters-БД на сети ресторанов, агрегирующий ETL (например, через Airbyte/dbt). Подходит для крупной сети. Нерационально для <5 точек.

## Что в v4 закладываем «на потом», чтобы не переделывать

Чтобы будущий sync можно было добавить дёшево, прямо сейчас:

1. **UUID-ключи везде** — без auto-increment integer. Это уже сделано (см. [05-DATA-MODEL.md](05-DATA-MODEL.md)). Конфликтов id между ресторанами не будет.
2. **`updated_at` колонка на каждой таблице** — для дельта-логики. Уже есть в схеме.
3. **`tenant_id` (restaurant_id) на каждой строке** — уже есть.
4. **Soft-delete где осмысленно** (`deleted_at` вместо физического DELETE) — упрощает sync. Внедрить как best-practice, но не обязательно.
5. **`idempotency_keys`** — для повторных push'ей в облако. Уже добавлено в v4 для HTTP-идемпотентности; ту же таблицу можно переиспользовать.
6. **Event log опционально:** если событий бизнес-логики окажется много, можно завести `domain_events` таблицу — но без неё легко обойтись, делая GORM-хуки.

Эти решения **бесплатны для v4** (одинаково правильны и без cloud), и **окупятся** когда понадобится cloud.

## Что НЕ закладываем сейчас

- Триггеры на `sync_log` — добавим в момент, когда понадобится.
- Конфиг `cloud.*` — не нужен, нет смысла.
- HTTP-клиент к Supabase / PostgREST — нет.
- Owner Dashboard — отдельный продукт, отдельный проект.

## Когда вернётся cloud — план миграции

1. Завести `cloud/` подпапку в репо: миграции для cloud-Postgres, конфиг.
2. В `server/internal/cloud/` написать `pusher.go` (читает `sync_log` или `domain_events`, шлёт в cloud).
3. Заполнить cloud начальным snapshot из локального `pg_dump` (или RPC `seed`).
4. Включить sync через флаг конфига.
5. Развернуть Owner Dashboard (Next.js на Vercel) поверх cloud Postgres.

Оценка работы: 2–3 недели на one-way push (Вариант A) при готовой v4-кодовой базе.

## Что НЕ переносим из v1, даже когда вернёмся в cloud

- `lib/runtime-mode.ts` — разграничение local/cloud режимов на фронте, легаси.
- Supabase auth для кассиров/официантов — не нужно, auth локально (PIN).
- Supabase Realtime — у нас уже есть свой SSE-hub.
- Supabase Storage — не используется в v4-flow.

## Лицензирование (если станет cloud)

Если cloud вернётся, появляется естественное место для license-сервера (как в v1, но переписать чище). До тех пор — license проверяется локально по подписанному файлу, без обращения к интернету (см. [06-BUSINESS-LOGIC.md](06-BUSINESS-LOGIC.md) раздел License).
